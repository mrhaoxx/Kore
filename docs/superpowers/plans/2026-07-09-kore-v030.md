# Kore v0.3.0：调度语义补全与观测性（Plan 6）Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** ① 调度器把在途建池预占当作已有池跟随（根治并发建池分裂）；② 池在线扩缩容（改 pool-size 后 agent 经 NRI UpdateContainers 原地调整全部成员，零重启）；③ agent Prometheus 指标 + `kubectl-kore` CLI。发布 v0.3.0。

**Architecture:** 全部在既有四组件内扩展。扩缩容以"晚于池创建时间的成员携带新 size"为触发（防陈旧注解回灌），成员容器 ID 在 plugin 层跟踪，resize 结果通过 CreateContainer 返回的 updates 广播。

**Tech Stack:** 新增 `github.com/prometheus/client_golang`（agent 指标）；kubectl 插件为独立 Go 二进制 `kubectl-kore`（controller-runtime client + text/tabwriter）。

## Global Constraints

- 继承 v0.1/v0.2 全部约束；镜像 v0.3.0
- 扩缩容规则：joiner 的 Pod `creationTimestamp` > 池 `CreatedAt` 时允许按 joiner 的 size 扩/缩；否则 size 不符仍 ErrConflict（防陈旧注解）；缩容从高位核收回；扩容按池的 numa 顺序就近增核（不够溢出走 preferred 语义）
- 调度器对"池存在但 size 更大"的 follower：容量需求 = 增量（TotalFree ≥ delta 即过）；size 更小/相等：零容量
- 指标端口 agent :9100 `/metrics`；命名前缀 `kore_`
- kubectl-kore 只读（nodes/pools/pod 三个子命令），走当前 kubeconfig context

### Task 2: 池在线扩缩容（allocator）

**Files:** Modify `pkg/allocator/pool.go`、`pool_test.go`

**Interfaces:**
- `PoolInfo` 增 `CreatedAt time.Time`
- `PoolRequest` 增 `PodCreated time.Time`
- `JoinPool` 语义扩展：存在且 size≠现有 → `req.PodCreated.After(p.CreatedAt)` 时调用内部 `resizePool`，否则 ErrConflict
- `resizePool(p *PoolInfo, newSize int) error`：缩容 `p.CPUs = dropLow 保留？` **从高位收回**（`dropHigh(p.CPUs, cur-newSize)`）；扩容按 `p.NUMA` 顺序逐 zone `pickInZone(zone, 尽量多, 1, pack)` 就近补足，不足再从其余 zone 距离序补（复用 pickPreferred 于剩余量）；扩容受 `checkSharedPoolMin` 约束；NUMA 列表按实际重算
- `NewState` 不变；restore 路径 `CreatedAt` 置零值（任何后来者可 resize）

**TDD 用例：**
- [ ] 晚创建成员带大 size → 池变大（同 zone 就近，成员共享新集合）；带小 size → 高位核归还共享池；早于池创建的 joiner 带异 size → ErrConflict；扩容触 sharedPoolMin → ErrInsufficient 且池不变；跨 zone 扩容 NUMA 列表更新
- [ ] 实现 → 回归 → Commit `feat: 分配器池在线扩缩容`
